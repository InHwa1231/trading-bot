# 퀀트 트레이딩 봇 개발 프로토콜 (토큰 및 리소스 최적화 모드)

이 파일은 프로젝트의 작업 효율과 토큰 비용 최적화를 위한 핵심 지침입니다. 모든 모델은 '로컬 컴퓨터 자원 우선 활용' 원칙을 준수해야 합니다.

## 📊 로컬 데이터 분석 우선 원칙 (Local Analysis First)
클로드는 방대한 로우 데이터(CSV, Log)를 직접 읽지 않는다. 분석이 필요할 경우 다음 절차를 따른다.
1. **분석 스크립트 제공:** 사용자가 로컬에서 실행하여 결과를 요약할 수 있는 Python 스크립트를 먼저 작성하여 제공하라.
2. **요약본 수신:** 사용자가 스크립트를 실행해 얻은 통계치(수익률, 승률, MDD, 에러 빈도 등)나 필터링된 요약 텍스트만 컨텍스트로 받아 분석하라.
3. **직접 읽기 금지:** 1MB 이상의 로그 파일이나 1,000줄 이상의 데이터 파일은 절대 직접 읽지 마라.

## 🤖 모델별 역할 정의 및 행동 수칙

### 1. Research & Analysis Mode (Target: Claude 4.5 Haiku)
- **주요 작업:** 웹 검색, API 문서 분석, 전략 요약.
- **행동 지침:**
  - 분석 결과는 `docs/research/` 폴더에 마크다운(.md) 파일로 저장하라.
  - 작업 완료 후 사용자에게 **"/clear 입력 후 Opus 모델로 전환"**을 제안하라.

### 2. Strategy & Logic Build Mode (Target: Claude 4.6 Opus)
- **주요 작업:** 퀀트 전략 설계, 복잡한 수학적 로직 및 백테스트 아키텍처 수립.
- **행동 지침:**
  - 로직 검증이 필요할 경우, 전체 데이터를 읽으려 하지 말고 **검증용 로컬 시뮬레이션 스크립트**를 작성해 사용자에게 주어라.
  - 설계 완료 후 사용자에게 **"/clear 입력 후 Sonnet 모델로 전환"**을 제안하라.

### 3. Development & Debugging Mode (Target: Claude 4.6 Sonnet)
- **주요 작업:** 실제 코드 구현, 에러 수정, 데이터 전처리 및 분석 스크립트 작성.
- **행동 지침:**
  - **로그 분석 요청 시:** 로그 전체를 복사하게 하지 말고, 특정 키워드(ERROR, 주문 번호 등)를 추출하거나 통계를 내는 파이썬 스크립트를 먼저 짜주어라.
  - **데이터 모순 방지:** 모든 데이터 처리 코드에는 결측치 처리 및 형변환(Float) 등 방어적 전처리 로직을 반드시 포함하라.
  - 수정 사항은 전체 코드가 아닌 변경된 부분(Diff)만 출력하라.

## 📉 토큰 절약 공통 규칙
- **최소 읽기:** `.claudeignore`에 등록된 파일(데이터, 로그)을 우회해서 읽으려 하지 마라.
- **출력 제한:** 터미널 결과는 핵심만 50줄 이내로 요약하라.
- **세션 관리:** 대화가 깊어지면 사용자에게 `/compact`나 `/clear`를 적극 권장하라.

## 📂 프로젝트 구조
- `/src`: 트레이딩 봇 소스 코드
- `/docs/research`: 전략 및 API 분석 문서
- `/scripts/analysis`: 로컬 데이터 분석용 유틸리티 스크립트
